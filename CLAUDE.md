# Qualify — Project Notes for Claude

> All decisions below are discussable. This project is in active early development and the architecture is still being shaped.

## Stack

- **Frontend**: React + TypeScript + Shadcn UI (Vite) — runs on port `65432`
- **Backend**: Python + FastAPI — runs on port `65444`

## Architecture Overview

Qualify is a local-first PaaS engine. The orchestration control plane runs on the developer's local machine. Remote servers are strictly container execution environments.

## Secret & Config Storage

### Local machine

- **`~/.config/qualify/state.json`** — non-sensitive config only:
  - Server IPs, ports, SSH users
  - Project names, git URLs, branches, domains, ports
  - Infra inference results (which services were detected)
  - Deployment history (metadata, not secrets)
  - Docker Secret *names* (references only, not values)

- **OS Keyring** (via `keyring` lib → Secret Service API on Linux, Keychain on macOS, Credential Manager on Windows) — one entry per server:
  - Sudo password for that server (needed for bootstrapping, UFW, etc.)

- **No env vars** for app config. No separate config files. state.json + keyring is the full local persistence story.

### On the server(s)

- **Docker Secrets** (Swarm encrypted Raft log) — all runtime secrets:
  - Database passwords (generated by Qualify during provisioning)
  - User-supplied API keys, tokens from `.env.template`
  - Injected into containers at runtime, never leave the server

### SSH authentication

- SSH private key path is stored in local state.json (reference only, e.g. `~/.ssh/id_rsa`)
- No key material is copied or stored by Qualify

## Secret Workflow

1. User fills in secret values in the UI (or Qualify generates them for provisioned infra)
2. Qualify SSHes to the server and runs `docker secret create <name> <value>`
3. Local state stores only the secret *name* so docker-compose can reference it
4. Actual secret values live encrypted in Docker Swarm's Raft log on the server

## Key Design Principles

- Local state is safe to sync, backup, or commit — it contains no secrets
- Servers are treated as dumb execution environments — no control plane software installed on them
- Docker Secrets is the server-side secret manager — no separate Vault, no .env files on disk

## Environments

Projects have multiple **environments** (e.g. dev, staging, prod). Environments are first-class — not an afterthought.

- Each environment belongs to a project and targets a specific server
- Each environment has its own: `server_id`, `domain`, `port`, `env_vars`, Docker Secret names, `inferred_infra`, deploy history
- The same server can host multiple environments (e.g. `dev.myapp.com` and `myapp.com` on the same VPS)
- Deployments target an **environment**, not a project+server pair directly
- Project holds only shared config: git URL, branch, Dockerfile path, build strategy

**No local execution mode.** Mac compatibility issues + complexity not worth it. Instead:
- Users deploy to a "dev" environment first, verify, then deploy to "prod"
- Qualify supports this flow natively through environment promotion

## Domain & DNS

- **Subdomain per environment** — path-based routing breaks apps that don't know their base path
- Traefik handles routing + Let's Encrypt SSL automatically once DNS resolves to the server

### DNS setup (two modes)

**Manual (default):** Qualify detects the server's public IP and shows: "Point `dev.myapp.com → <IP>` in your DNS panel, then continue."

**Auto via Cloudflare (optional):** User provides a Cloudflare API token. Qualify creates the A record automatically when an environment is created.
- Token stored in **OS keyring** locally (never in state.json)
- Other providers (Route53, Namecheap) are future work — Cloudflare first
- Flow: create environment → Qualify calls Cloudflare API → A record created → Traefik picks up domain → Let's Encrypt issues cert

## Env Var Pre-flight

Before every deploy, Qualify runs an env var check:
1. Diff `.env.template` keys against what the environment has configured
2. Categorize each var: ✓ provided / ⚡ will be provisioned (postgres, redis) / ✗ missing
3. Hard block on missing vars with no default, warn on optional ones
4. User can acknowledge warnings and proceed, but missing required vars must be filled first

## Why Not Just GitHub Actions?

This question comes up often. The key differences:

**GitHub Actions requires:**
- An external registry: DockerHub (1 free private repo), GHCR, ECR — all need account setup and credential management
- Secrets configured per-repo in GitHub (SSH keys, registry tokens)
- A workflow YAML file maintained per project
- Dependency on GitHub's availability and build runners
- Code passing through a third-party build environment

**Qualify instead:**
- **Provisions a private registry on your own server** (`registry:2` behind Traefik with basic auth) — no external registry account ever needed
- **Local build** uses the developer's CPU/RAM — keeps cheap VPS lean (a $6/mo server doesn't need to build anything)
- **Remote build flag** (`--remote`) for when the local machine is weak and server is beefy
- **Works with any git host** — GitLab, Gitea, Bitbucket, bare repos; no GitHub dependency
- **Zero workflow files** — no YAML to maintain, no per-project CI setup
- **Code stays on your infrastructure** — never touches third-party build runners

**One-liner:** Qualify gives solo devs and small teams a Heroku-style `deploy` experience on their own servers, without configuring CI pipelines, managing registry accounts, or installing anything on the server upfront.

## Distribution

- Distributed as a **single PyInstaller binary** — no Python, Node.js, or any runtime required on the user's machine
- Binaries built via GitHub Actions on release tags: `qualify-linux-x86_64`, `qualify-macos-arm64`, `qualify-windows-x86_64.exe`
- Frontend is bundled inside the binary via PyInstaller `--add-data`
- Backend serves the bundled frontend static files — single process, single port (`65432`)
- In dev mode: frontend runs on `65432` (Vite), backend on `65444` (uvicorn), Vite proxies `/api`

## Windows Compatibility

The tool runs **natively on Windows** — no WSL required. All Linux-specific commands (`systemctl`, `ss`, `ufw`, etc.) run on the remote server over SSH, never locally. Local requirements are just Docker Desktop + Git, both available on Windows.
